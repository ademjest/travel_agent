# 旅游风险规划 QQ Bot

这是一个面向自驾旅行的 QQ 群 Agent。当前版本已经接入高德 Web 服务和 OpenAI 兼容大模型，可通过固定指令或自然语言查询天气、路线和路况，也可由 LLM 在当前用户权限范围内查看、刷新、确认、修改和取消预约提醒。

## 已实现功能

- QQ 群内 `@机器人` 接收和回复消息
- 指定 QQ 群白名单
- 中文地点地理编码
- 实时天气查询
- 最多四天的天气预报
- 高德推荐驾车路线
- 路线距离、实时预计耗时、收费和红绿灯信息
- 路线分段 `畅通/缓行/拥堵/严重拥堵` 里程和数据覆盖率汇总
- 具体拥堵路段的道路名、导航说明、沿路线近似里程和首尾坐标
- 基于覆盖率约束的车辆拥堵风险等级
- 自然语言问题理解
- 受约束的 ReAct 工具调用循环
- 单轮识别多个意图，并调用多个相互独立的白名单工具
- 最多四轮模型决策、六次工具调用，避免无限执行
- 工具调用轨迹日志，不记录或展示隐藏思维链
- LLM 失败时固定指令仍可继续使用
- 按群成员隔离的最近 6 轮对话记忆
- SQLite 本地持久化，重启后短期对话仍可恢复
- 消息 `processing/completed/failed` 状态、失败重试和已生成回复复用
- 群内一次性绑定码和 QQ 私聊文档上传
- `.docx/.txt/.md/.xlsx` 旅行文档长期保存和相关片段检索
- 重复文档 SHA-256 去重
- 群聊单图识别景点预约规则，并生成待确认预约计划
- 为整份预约草稿选择一份最佳匹配行程文档，不跨版本混用日期
- 默认双提醒、自定义提醒、修改和取消预约提醒
- SQLite 持久化定时任务，通过 Outbox 在重启和发送失败后继续投递
- 事件租约续期、最多 8 次发送重试和死信隔离
- 可配置数据留存、孤儿预约图片清理、后台任务自动重启和 OneBot 健康检查

使用的高德官方接口：

- [地理编码](https://lbs.amap.com/api/webservice/guide/api/georegeo)
- [天气查询](https://lbs.amap.com/api/webservice/guide/api/weatherinfo)
- [路径规划 2.0](https://lbs.amap.com/api/webservice/guide/api/newroute)

## 1. 环境要求

- Python 3.10 或 3.11 推荐
- 已创建并配置 QQ Bot
- 高德开放平台账号
- 高德 Web 服务类型的 Key
- 支持 OpenAI Chat Completions、原生 tool calling 和图片输入的多模态模型接口

你已经使用 Conda 环境 `agent`，不需要再创建项目内 `.venv`：

```powershell
conda activate agent
cd E:\Agent\travel_agent-auto-refresh
pip install -r requirements.txt
```

## 2. 创建高德 Web 服务 Key

1. 登录[高德开放平台控制台](https://console.amap.com/dev/key/app)。
2. 创建一个应用，例如 `青甘自驾风险Bot`。
3. 在应用中添加 Key。
4. Key 的服务平台选择 **Web服务**，不是 Android、iOS 或 Web端 JS API。
5. 保存生成的 Key。

天气、地理编码和路径规划可以共用这个 Web 服务 Key。实际额度和权限以高德控制台当前显示为准。

## 3. 配置环境变量

本项目调用 `load_dotenv()`，从 `E:\Agent\travel_agent-auto-refresh` 启动时会向父目录查找，因此可以继续使用已有的：

```text
E:\Agent\.env
```

在该文件中确保存在：

```env
QQ_BOT_APPID=你的QQ机器人AppID
QQ_BOT_SECRET=你的QQ机器人AppSecret
QQ_BOT_ALLOWED_GROUPS=你的旅行群group_openid
QQ_BOT_ALLOW_ALL_GROUPS=false
AMAP_API_KEY=你的高德Web服务Key
LLM_API_KEY=你的模型API密钥
LLM_BASE_URL=你的OpenAI兼容接口地址
LLM_MODEL_ID=你的模型ID
```

`QQ_BOT_ALLOWED_GROUPS` 为空时默认拒绝所有 QQ 官方群。仅本地开发临时联调时才可设置 `QQ_BOT_ALLOW_ALL_GROUPS=true`；生产环境必须使用明确白名单。

不要把真实 Key 写进 `.env.example`。`.env.example` 只用于记录配置项，可以提交到 Git；真实 `.env` 不能提交。

修改环境变量后，需要重启机器人进程才能生效。

## 5. 群聊指令

机器人支持两种入口：

- 在群输入框输入 `/`，从 QQ 原生指令面板选择指令。QQ 会自动带上机器人，无需手动输入 `@机器人`。
- 直接在群里 `@机器人` 后输入完整指令。

### 配置 QQ 原生“/”指令面板

指令面板不能由机器人代码在运行时自动注册，需要在 [QQ 开放平台](https://bot.q.qq.com/open) 手动配置：

1. 打开对应机器人，进入 `发布设置 → 功能配置 → 指令配置`。
2. 添加下表中的指令，权限选择“所有用户”。
3. 使用场景至少勾选 `QQ群`；如果也希望私聊使用，可同时选择消息列表单聊。
4. 常用指令建议按下表顺序排列，然后按开放平台当前流程保存、自测并上线。

| 指令名 | 指令介绍 |
| --- | --- |
| 帮助 | 查看快捷操作菜单 |
| 状态 | 检查机器人配置 |
| 查询天气 | 查询当前天气 |
| 天气预报 | 查询未来天气 |
| 查询路线 | 规划驾车路线 |
| 查询路况 | 查看实时拥堵 |
| 上传文档 | 导入旅行资料 |

QQ 官方当前允许最多配置 24 个指令；指令名不超过 8 个中文字符，介绍不超过 15 个中文字符。开放平台只负责在客户端展示入口，具体响应仍由本项目代码实现。

发送“帮助”“菜单”“旅行面板”或“状态”时，机器人会返回带快捷按钮的 Markdown 面板。群聊按钮会把 `@机器人 + 指令` 填入输入框，但 QQ 不允许群聊按钮自动发送；天气、预报、路线和路况仍需补充地点后手动确认发送。

检查配置：

```text
@旅游bot 状态
```

实时天气：

```text
@旅游bot 查询天气 西宁
@旅游bot 查询天气 青海湖
```

天气预报：

```text
@旅游bot 天气预报 茶卡盐湖
@旅游bot 天气预报 敦煌
```

驾车路线：

```text
@旅游bot 查询路线 西宁 -> 青海湖二郎剑景区
@旅游bot 查询路线 青海湖 -> 茶卡盐湖
```

路线与分段路况：

```text
@旅游bot 查询路况 西宁 -> 青海湖
@旅游bot 查询路况 张掖七彩丹霞 -> 敦煌
```

路线指令推荐使用 `->` 分隔起点和终点，避免复杂景点名称被错误拆分。也支持：

```text
查询路况 西宁 到 青海湖
查询路线 西宁 青海湖
```

LLM 配置完成后，还可以直接使用自然语言：

```text
@旅游bot 西宁现在天气怎么样？
@旅游bot 明天青海湖天气适合游玩吗？
@旅游bot 从西宁开车去青海湖大概要多久？
@旅游bot 现在从青海湖到茶卡盐湖堵不堵？
@旅游bot 帮我综合看看明天从西宁去青海湖是否适合自驾
```

当信息不足时，Agent 会追问。例如“帮我看看路况”没有起点和终点，不会猜测地点。

### 上传长期旅行资料

QQ 群文件是独立消息，不能和 `@机器人` 合并发送，因此当前版本使用“群内申请绑定码 + 私聊发送文件”的方式导入文档。

第一步，在目标群中发送：

```text
@旅游bot 上传文档
```

机器人会返回类似下面的一次性绑定码：

```text
QG-ABC234
```

第二步，打开与机器人的私聊，在 10 分钟内先发送绑定码。机器人回复“绑定成功”后，再单独发送文件，不需要附加文字。

当前支持：

- `.docx`
- `.txt`
- `.md`
- `.xlsx`

单个文件最大 5 MB。旧版 `.doc` 和 `.xls` 暂不支持，请分别转换为 `.docx` 和 `.xlsx`。

Excel 会导入所有可见且非空的工作表，包括单元格文本、数字、布尔值、日期和文件中已保存的公式结果；隐藏工作表、图表、图片、批注和格式不会进入知识库。

每个绑定码只能由一个私聊用户兑换，并在收到一次附件消息后失效，无论文件是否解析成功。同一个私聊用户兑换新绑定码时，之前尚未使用的绑定会失效。绑定码只以 SHA-256 哈希保存在 SQLite 中，不保存明文。

机器人会在私聊中回复提取到的字符数和文档片段数。文档保存在绑定群的共享长期资料中，不受最近 6 轮对话限制。后续回到群里询问：

```text
@旅游bot 文档里8月18日住在哪里？
@旅游bot 我们计划哪天去敦煌？
@旅游bot 根据上传的行程，明天这一段路天气如何？
```

文档附件下载地址通常具有时效性，因此机器人收到消息后会立即下载、解析并写入本地数据库，不依赖之后再次访问 QQ 文件地址。

如果 LLM 已配置，上传时还会提取一份不超过约 1800 字的长期摘要，重点保存旅行日期、每日路线、住宿、集合时间、车辆限制、确认事项和待确认事项。后续每次文档问答会优先提供与问题相关的原文片段，再用剩余上下文预算补充摘要；摘要失败不会影响原文保存。

### 景点预约图片与定时提醒

在允许的 QQ 群中先发送 `@机器人 制定预约`，机器人会进入 30 分钟的预约制定模式；随后发送一张景点预约攻略图片即可创建预约计划草稿。也可以在图片消息中直接填写 `制定预约`，一步启动并识别。普通图片不会再自动创建预约计划，为后续扩展通用识图功能保留独立入口。发送 `退出制定预约` 可以主动结束当前流程。

攻略图片支持 JPEG、PNG 和 WebP，单张最大 5 MB；一次发送多张图片会被拒绝，请逐张处理。附件必须提供有效的 HTTPS 下载地址，机器人收到后会立即下载，不能依赖之后继续访问 QQ 的临时链接。识图或下载失败时预约制定模式保持有效，可以修正图片后直接重试；成功生成草稿后模式自动结束。

处理流程如下：

```text
制定预约命令或活跃预约制定模式
→ 单张预约攻略图片
→ 保存原图并按 SHA-256 去重
→ 多模态模型提取景点、价格、开放时间和预约规则
→ 在群共享旅行文档中匹配明确游览日期
→ 生成待确认草稿
→ 创建者补充或修改
→ 明确确认后才创建正式提醒
```

模型只提取图片中明确存在的景点和预约规则。游览日期由 Python 从群共享资料中选择一份覆盖景点最多的最佳匹配行程文档，并仅在该文档内确定性匹配；同分时选择最新文档，不会跨行程版本拼接日期。预约日期按规则确定性计算：`提前 N 天` 按自然日回退，`提前 N 月` 按自然月回退，目标月份没有对应日期时取该月最后一天。识别不到景点或日期时不会猜测，而是保留与原图关联的全手动草稿。

除固定命令外，也可以直接使用自然语言，例如“查看我的预约提醒并把 A-000123 改到 8 月 21 日”“刷新 R-20260722-001，然后确认”。LLM 只能看到本轮意图允许的工具，平台、群和创建者身份由程序从当前事件注入，模型不能自行指定或越权修改。当前“创建预约提醒”仍以已经上传图片生成的草稿为起点；不会脱离现有草稿凭空创建计划。

常用命令：

```text
制定预约
退出制定预约
补充预约 R-20260722-001 2 2026-08-20
新增预约 R-20260722-001 莫高窟 2026-08-20 提前1月
新增预约 R-20260722-001 黑独山 2026-08-22 无需预约
设置提醒 R-20260722-001 1 2026-08-15 07:30
设置提醒 R-20260722-001 1 2026-08-14 20:00, 2026-08-15 07:30
刷新预约 R-20260722-001
确认预约 R-20260722-001
取消预约 R-20260722-001
查看预约提醒
修改预约提醒 A-000123 游览日期 2026-08-21
修改预约提醒 A-000123 时间 2026-07-20 20:00, 2026-07-21 07:30
取消预约提醒 A-000123
```

`查看预约提醒` 会先使用群内最新行程文档刷新当前用户的未确认草稿；`确认预约` 在确认前执行相同刷新。也可以点击帮助或状态面板中的静态“刷新预约”按钮，按钮会填入 `刷新预约 `，补充计划编号后发送。刷新只处理 `needs_input` 和 `not_scheduled` 项目，不覆盖手工补充日期、已就绪项目或已确认计划。因此先发送预约图片、后上传 Excel 时，不需要重新发送图片。

需要预约且没有自定义时间时，默认在建议预约日前一天 `20:00` 和预约日当天 `09:00` 提醒。自定义的一个或多个完整时间会替换两条默认提醒。群内显示使用 `Asia/Shanghai`，数据库统一保存 UTC 时间。无需预约和无需提前统一按“无需预约”存档，不创建提醒。

计划必须由创建者明确确认；确认前不会发送提醒。只有创建者可以查看、确认、修改或取消自己的计划和提醒。机器人不会自动代用户预约，也不会自动搜索或猜测官方链接；预约渠道为空时会提示前往景区官方渠道核对。

到期消息通过已有 SQLite Outbox 发送。机器人离线期间错过、但游览日期尚未过去的提醒，会在下次启动时标记为延迟补发；游览日期已经过去的提醒会过期，不再发送；群被移出允许列表后会阻止发送。平台发送失败只重试已生成消息，不会再次识图、调用模型或重复创建提醒。

预约图片会发送到 `LLM_BASE_URL` 配置的外部多模态 API。原图保存在 `data/images/`，SQLite 保存文件路径、提取结果和提醒状态；`data/` 已排除在 Git 之外。日志不记录图片 URL、base64、完整 OCR 文本、密钥或群内私密内容，只记录哈希前缀、大小、模型、耗时和状态。

`requirements.txt` 已加入 `openpyxl`；`requirements-onebot.txt` 通过 `-r requirements.txt` 自动继承，无需重复添加。`.github/workflows/scheduled-bot.yml` 无需增加缓存路径，因为现有加密归档会打包整个 `data/` 目录，数据库与 `data/images/` 会一起加密保存；不能把原图作为明文 GitHub Artifact 上传。

## 6. 返回结果说明

天气接口使用地理编码结果中的行政区 `adcode` 查询，因此高德返回的是景点所属行政区天气，不一定是景点坐标处的微气候。例如“青海湖”可能匹配到周边行政区，实际自驾前仍需结合更细粒度天气源和官方预警。

路线查询使用高德推荐策略 `strategy=32`。路况查询请求 `cost,tmcs` 字段，并汇总路线中的：

- 畅通
- 缓行
- 拥堵
- 严重拥堵

如果高德返回了 TMC 坐标，机器人还会列出最多 6 个具体拥堵路段，包括父级道路名、导航说明、沿整条路线的近似起止里程、拥堵长度和首尾坐标。相邻且同道路、同状态的路段会自动合并。近似里程由各 step/TMC 返回距离累计得到，坐标和高德实时导航更适合作为最终定位依据。

机器人同时显示 TMC 已覆盖里程占总路线的比例。覆盖率低于 70% 时，即使已返回部分没有明显拥堵，也不会给出“低风险”结论，而会提示数据覆盖不足。

“车辆拥堵风险”只根据交通速度和拥堵里程计算，不代表道路是否封闭，也不能识别施工、落石、积雪、结冰等危险，不能替代交警、交通运输部门和景区公告。

## 7. Agent 工作方式

固定指令优先直接调用确定性服务，不产生 LLM 费用。无法匹配固定指令的自然语言消息进入 Agent：

```text
用户问题
→ LLM 判断是否缺少信息
→ 选择白名单工具
→ Python 执行一个或多个旅行/预约工具
→ 工具结果返回 LLM
→ LLM 基于真实数据生成结论
```

当前工具：

- `get_current_weather`
- `get_weather_forecast`
- `get_driving_route`
- `get_route_traffic`
- `list_reservation_plans`
- `refresh_reservation_plan`
- `confirm_reservation_plan`
- `cancel_reservation_plan`
- `complete_reservation_item_date`
- `add_reservation_item`
- `set_reservation_reminder_times`
- `modify_reservation_item_date`
- `modify_reservation_item_times`
- `cancel_reservation_item`

Agent 最多执行四轮模型决策、六次工具调用；同一轮可以同时请求天气与路况、查看计划与修改项目等多个独立工具。相同工具和参数会复用本轮缓存。实时查询和预约写操作如果没有实际完成对应工具调用，模型的直接文字声明会被拒绝。单次模型请求超时为 90 秒，SDK 最多自动重试一次。

```text
Agent tool trace: get_route_traffic({'origin': '西宁', 'destination': '青海湖'})
```

系统不会输出模型隐藏思维链。天气和路况必须来自工具；安全结论必须说明高德行政区级天气和缺少道路封闭公告等数据局限。

### 对话与文档记忆

每个 `群 + 成员` 独立维护最近 6 轮用户/Bot 对话，避免成员之间的临时上下文互相污染。超过 6 轮的普通对话不会继续发送给 LLM，但仍保存在 SQLite 中，为后续摘要和检索功能预留数据。

每个 `群 + 成员` 仍最多读取最近 6 轮对话，但实际发送给模型的历史内容限制为约 3000 字符。

旅行文档使用另一套长期记忆：保存文档目录和分块原文，通过 SQLite FTS5 索引筛选候选片段，并为“西宁、敦煌”等二字词保留数据库内子串检索降级。Python 每次只对少量候选精排，不再加载群内全部分块。最终最多提供 2 个相关片段，总文档上下文最多约 3200 字符；相关片段优先使用预算，摘要只占用剩余空间。如果没有明显关键词匹配，会回退到最近上传文档的开头片段。

数据库默认位置（相对项目目录）：

```text
data/travel_bot.db
```

`data/` 已加入 `.gitignore`，因为其中可能包含群成员对话、行程、住宿、预约提取结果和原始图片等敏感信息。GitHub Actions 的状态检查点会把整个目录加密后缓存，因此 SQLite 和 `data/images/` 会一起恢复。

## 9. 常见问题

### 状态显示未配置 AMAP_API_KEY

确认 Key 已写入 `E:\Agent\.env`，然后完全停止并重新运行 `python bot.py`。

### 高德返回 INVALID_USER_KEY

检查 Key 是否正确，并确认创建的是 **Web服务** 类型 Key。

### 找到了错误地点

如果高德返回多个不同候选，机器人不会再静默选第一个，而是列出候选并要求补充省、市或区县。请使用更完整的地点名称，例如：

```text
查询天气 青海省海南州青海湖二郎剑景区
查询路线 青海省西宁市 -> 青海湖二郎剑景区
```

### 路况没有分段数据

部分地区或道路可能不返回 `tmcs`。机器人仍会显示高德根据当前交通状况计算的预计耗时，但不会编造分段拥堵信息。

### Bot 收到消息但没有回复

检查终端是否出现高德接口错误。机器人每次请求会进行地理编码，模糊地点可能无法匹配；同时确认高德 Key 的额度尚未耗尽。

### 固定指令可用，但自然语言不可用

在群里发送“状态”，确认显示 `LLM Agent：已配置`。然后检查：

- `LLM_API_KEY`、`LLM_BASE_URL` 和 `LLM_MODEL_ID` 是否完整。
- 当前模型是否支持 Chat Completions 的原生 tool calling。
- 模型服务是否允许 `tools` 和 `tool_choice` 参数。

模型接口失败时，机器人会提示继续使用固定指令，不会影响高德查询。

### 上传文档后机器人没有识别

不要直接把文件发送到 QQ 群。请先在目标群发送 `@机器人 上传文档`，再私聊机器人完成以下两步：

1. 单独发送 `QG-XXXXXX` 绑定码，确认机器人回复“绑定成功”。
2. 单独发送 `.docx/.txt/.md/.xlsx` 文件，不要只发送文件名或腾讯文档页面链接。

终端收到私聊事件时会显示 `Received private message` 和 `attachments=<数量>`。如果文件消息显示 `attachments=0`，说明 QQ 没有向机器人下发该附件；请确认是在机器人私聊窗口中发送，并检查当前机器人账号是否已获得 C2C 消息能力。PDF、图片以及旧版 `.doc`、`.xls` 当前不会作为私聊旅行文档解析；景点预约图片使用群聊 `制定预约` 命令或活跃预约制定模式的独立入口。

绑定码有效期为 10 分钟，收到一次附件消息后立即失效。文件格式错误、解析失败、绑定过期或已使用时，都需要回到目标群重新发送“上传文档”。

## 10. 群聊全量消息授权说明

QQ 官方文档提供 `GROUP_MESSAGE_CREATE`：群主允许机器人接收群内全部消息后，平台可推送后续群消息。这个能力不等于读取历史记录，授权前、机器人入群前或离线期间的消息不会通过历史查询接口补回。

群主需要在 QQ 群中的机器人资料或机器人设置页面，开启类似“允许机器人接收群内全部消息”的开关。该开关与“允许机器人主动在群聊内发言”是不同权限，具体名称以当前 QQ 客户端页面为准。

但当前项目使用的 `qq-botpy 1.2.1`，以及我核对的 botpy 官方主分支，都还没有 `GROUP_MESSAGE_CREATE` 的事件 parser。因此目前即使群主开启授权，项目代码也只处理 `@机器人` 消息。后续实现全量群消息时需要：

1. 保留 `public_messages=True` intent。
2. 为 botpy 增加 `GROUP_MESSAGE_CREATE` payload 解析兼容层。
3. 实现消息 `msg_id` 去重。
4. 普通群消息只做本地相关性过滤和存储，不触发 LLM、不自动回复。
5. 在群内明确告知成员消息保存范围和清理方式。

## 11. 可靠性、上下文和隐私

机器人回复会先写入 SQLite outbox，再调用 QQ 或 OneBot 发送接口。发送失败只重试已经生成的 payload，不会重新调用 LLM、高德或文档解析。连续失败 8 次后消息进入死信状态，不再阻塞后续消息。长时间识图、解析文档或调用 LLM 时会续租事件处理权；预约草稿、手工新增项目和群上传绑定码使用事件幂等键，崩溃重放不会重复创建。

群聊上下文采用固定预算：引用消息最多 800 字、最近观察到的群消息最多 2200 字、相关文档最多 3200 字，完整上下文快照不超过约 7000 字。上传文档和群聊消息会放入非可信资料边界，不能修改系统身份、规则或工具权限。日志只记录决策类型、工具名、耗时和上下文大小，不记录隐藏思维链。

QQ 官方 Bot 只能看到平台实际推送给机器人的消息，因此上下文会明确标记为“部分群消息”。留存清理在启动时执行，并由受监督后台任务每日执行。默认策略为：群聊 30 天、终态事件/Outbox 30 天、绑定码 7 天、已取消预约与孤儿图片 90 天；文档默认永久保留。可通过 `RETENTION_*_DAYS` 环境变量调整，`RETENTION_DOCUMENT_DAYS=0` 表示不自动删除文档。

手动执行当前留存策略：

```bash
python -c "from maintenance import MaintenanceService; from memory_store import MemoryStore; print(MaintenanceService(MemoryStore(), 'data/images').run_once())"
```

## 12. 可选 NapCat/OneBot 部署

NapCat 是非 QQ 官方协议，只适合实验性部署。它需要长期保存 QQ 登录会话、设备身份和配置文件，因此不能部署在 GitHub-hosted Actions。请使用持久 Linux VPS、NAS 或家庭服务器，并使用专用、低价值 QQ 账号；非官方协议可能触发 QQ 风控、限制登录或封禁账号。

官方 Bot 和 NapCat 不要同时在同一个群中作为回复者运行。切换到 NapCat 前，应先从官方 Bot 的允许群列表中移除该群，避免重复回复和事件状态分裂。

### 部署步骤

1. 在 Linux 主机安装 Docker Engine 和 Compose 插件，克隆仓库。
2. 创建部署配置：

```bash
cd deploy/napcat
cp .env.example .env
openssl rand -hex 32
openssl rand -hex 32
```

将两个不同的随机值分别写入 `ONEBOT_ACCESS_TOKEN` 和 `ONEBOT_INBOUND_TOKEN`，再填写群号、高德和 LLM 配置。不要提交 `.env`。

3. 构建并启动：

```bash
docker compose --env-file .env up -d --build
docker compose ps
```

NapCat 镜像和 Python 基础镜像均固定了版本与 digest。`qq_config`、`napcat_config` 和 `travel_data` 是持久卷。

4. NapCat WebUI 只绑定服务器的 `127.0.0.1:6099`。从本机建立 SSH 隧道：

```bash
ssh -L 6099:127.0.0.1:6099 your-user@your-server
```

然后在本机访问 `http://127.0.0.1:6099` 完成专用 QQ 登录。

5. 在 NapCat 中配置 OneBot：

- HTTP 服务监听容器端口 `3000`，Access Token 使用 `ONEBOT_ACCESS_TOKEN`。
- HTTP 事件上报地址为 `http://onebot:8000/onebot`。
- 上报请求使用 `Authorization: Bearer <ONEBOT_INBOUND_TOKEN>`；也支持 `X-OneBot-Token`。
- 只启用确实需要的群，并确保群号存在于 `ONEBOT_ALLOWED_GROUPS`。

普通群消息只保存为上下文，不调用 LLM、不自动回复。`@机器人`，或事件中能够确认是回复机器人本身的消息，才会进入 Agent。所有允许群消息都只保存归一化文本和必要标识，不保存完整 OneBot 原始 JSON。

OneBot 服务提供 `GET /health`，返回后台任务、Outbox、死信、预约提醒和留存任务状态，不返回 Access Token、入站 Token 或 API Key。

### 备份与回退

先生成一致的 SQLite 备份：

```bash
docker compose exec onebot python -c "import sqlite3; s=sqlite3.connect('/app/data/travel_bot.db'); d=sqlite3.connect('/app/data/travel_bot.backup.db'); s.backup(d); d.close(); s.close()"
docker compose cp onebot:/app/data/travel_bot.backup.db ./travel_bot.backup.db
openssl enc -aes-256-cbc -pbkdf2 -salt -in travel_bot.backup.db -out travel_bot.backup.db.enc
rm travel_bot.backup.db
```

备份密码、QQ 凭据、OneBot token 和 API Key 只能保存在部署环境或密钥管理器中。需要回退到官方 Bot 时，先停止 NapCat 的事件上报和回复服务，再恢复官方 Bot 的群白名单。
