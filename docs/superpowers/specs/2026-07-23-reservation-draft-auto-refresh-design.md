# 预约草稿按需自动刷新设计

## 背景

当前预约草稿只在群内预约图片首次生成时读取行程文档。如果用户先发送图片、后通过私聊上传 Excel，已有草稿不会重新匹配行程日期。`查看预约提醒` 因而继续显示旧的 `needs_input`、候选日期或 `not_scheduled` 状态。

另一个问题是自然语言回复可能建议用户发送不存在的“确认创建预约提醒”。当前确定性命令实际要求 `确认预约 R-YYYYMMDD-NNN`，错误短语会进入普通 LLM，而 LLM 没有预约写入工具。

## 目标

- 查看或确认草稿前，按需使用最新群共享行程文档刷新未确定项目。
- 提供显式 `刷新预约 R-YYYYMMDD-NNN` 命令。
- 在 QQ 帮助/状态键盘中增加静态“刷新预约”按钮，填入命令前缀。
- 不覆盖手工日期、已就绪项目或已确认计划。
- 阻止“确认创建预约提醒”进入普通 LLM 并产生误导。

## 非目标

- 不在上传 Excel 的私聊事务中立即修改预约草稿。
- 不让 LLM 获得创建、确认、修改或取消预约的副作用工具。
- 不改变已确认计划和已生成提醒。
- 不新增动态的逐计划 QQ 按钮。
- 不解决多个候选日期的自动选择问题。

## 用户流程

### 默认按需刷新

用户上传新的 `.xlsx` 行程后，在群内发送：

```text
查看预约提醒
```

系统先刷新该用户在当前群内的所有 `draft`，再返回计划列表。用户发送：

```text
确认预约 R-20260723-001
```

系统先刷新目标草稿，再执行现有确认校验。仍有 `needs_input` 项目时继续拒绝确认并要求补充日期。

### 显式刷新

新增命令：

```text
刷新预约 R-20260723-001
```

系统刷新目标草稿并返回完整草稿。回复应说明实际更新数量；没有变化时明确说明未找到可自动补齐的新日期。

### 静态按钮

帮助/状态键盘的预约行增加：

```text
🔄 刷新预约
```

按钮数据为 `刷新预约 `，使用现有填入输入框但不自动发送的行为。用户补充计划编号后发送。OneBot 不渲染 QQ 键盘，但继续支持相同文本命令。

### 无效确认短语

精确短语：

```text
确认创建预约提醒
```

由确定性命令层处理，不调用 LLM，也不执行确认。回复指导用户先发送 `查看预约提醒` 获取计划编号，再发送 `确认预约 R-YYYYMMDD-NNN`。

## 刷新规则

刷新只作用于满足以下条件的项目：

- 所属计划状态为 `draft`。
- 计划属于当前平台、群和创建者。
- 项目要求预约。
- 项目当前状态为 `needs_input` 或 `not_scheduled`。
- 项目尚无确定的 `visit_date`。

服务一次读取群共享文档，并复用现有 `ReservationItineraryResolver`。该解析器继续采用“覆盖目标景点最多，同分选择最新文档”的单一行程来源规则。

每个项目按解析结果更新：

- 唯一日期：设置 `visit_date`，按现有规则计算 `booking_date`，候选日期设为该日期，状态设为 `ready`。
- 多个日期：清空确定日期，保存候选日期，状态设为 `needs_input`。
- 行程明确未安排：清空确定日期和候选日期，状态设为 `not_scheduled`。
- 未找到：清空确定日期和候选日期，状态设为 `needs_input`。

所有项目更新在一个 SQLite 事务中完成。刷新过程不得创建提醒；只有现有确认流程可以将草稿转为正式计划并写入提醒任务。

## 组件修改

### `commands.py`

- 增加 `刷新预约 <PLAN_CODE>` 正则和 `reservation_refresh` 命令。
- 增加“确认创建预约提醒”的确定性指导命令。

### `reservation_service.py`

- 增加刷新单个草稿的方法。
- `reservation_list` 在格式化前刷新当前用户的草稿。
- `reservation_confirm` 在校验前刷新目标草稿。
- `reservation_refresh` 显式刷新并返回草稿及更新结果。

### `memory_store.py`

- 增加原子更新草稿解析结果的方法。
- SQL 必须同时约束平台、群、创建者、计划状态和项目状态，避免修改其他用户或已确认数据。

### `qq_ui.py`

- 在预约按钮行增加静态“刷新预约”按钮。
- 更新帮助 Markdown，展示刷新命令。

### `travel_agent.py`

- 增加预约操作边界说明：预约状态修改必须使用确定性命令，不能声称已创建提醒或建议不存在的确认命令。

## 错误处理

- 计划不存在：返回“预约计划不存在”。
- 非创建者操作：沿用现有权限错误。
- 计划不是 `draft`：显式刷新返回不可刷新提示，不修改数据。
- 没有行程文档或没有匹配日期：保持安全状态并返回无变化提示。
- SQLite 更新失败：整个刷新事务回滚，原草稿保持不变。
- 刷新失败不得绕过现有确认校验。

## 测试

- 先创建图片草稿、后上传 Excel，`查看预约提醒` 自动补齐日期。
- `确认预约` 在确认前自动刷新并创建正确提醒。
- `刷新预约 R-...` 返回更新后的完整草稿。
- 静态按钮填入 `刷新预约 ` 且不自动发送。
- 手工补充日期和 `ready` 项目不被覆盖。
- 已确认或已取消计划不被刷新。
- 多候选日期保持 `needs_input`。
- `not_scheduled` 项目可在新行程明确安排后变为 `ready`。
- “确认创建预约提醒”返回确定性指导且不调用 LLM。
- 其他用户不能查看、刷新或确认计划。
- QQ 官方和 OneBot 文本命令行为一致。

## 验收标准

在截图对应的现有数据中，部署新版本后不需要重新上传图片。用户发送 `查看预约提醒` 时，旧草稿会读取已经保存的 Excel，并自动补齐能够唯一匹配的景点日期。随后使用精确的 `确认预约 R-...` 可以创建正式提醒；仍有歧义的项目必须继续人工补充。
